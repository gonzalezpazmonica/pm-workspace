## Qué hace este PR (en lenguaje no técnico)

Cuando los agentes de Savia colaboran en revisiones de código, el resultado final (aprobado o rechazado) dependía de quién sumaba más votos, sin ninguna forma objetiva de medir cuánto consenso había ni qué juez era el cuello de botella. Este cambio traduce cuatro principios del sistema Proto (Arc Institute, 2026) al flujo de trabajo interno de Savia.

El primero formaliza algo que ya ocurría implícitamente: las ramas `agent/*` son propuestas en proceso, y `main` es el estado validado. Se añade una guarda que impide usar una rama de propuesta como fuente de otra rama, evitando el error de construir sobre trabajo no revisado.

El segundo añade puntuación numérica al Tribunal de Revisión. Cada juez asigna un valor entre 0 y 1. Un script agrega esas puntuaciones, identifica qué juez bloqueó más y calcula si el consenso supera el umbral configurable. El resultado es auditable: ya no es solo "aprobado/rechazado" sino cuánto consenso hubo y dónde estuvo el desacuerdo.

El tercero establece un orden obligatorio en los workflows de múltiples etapas: primero las operaciones baratas (lint, compilación rápida), luego las medias (tests unitarios), por último las caras (tests de integración, generación de contexto completo). Esto evita desperdiciar tiempo de modelo en trabajos que un paso anterior habría bloqueado de todas formas.

El cuarto hace que el catálogo de skills sea descubrible por agentes externos. Un script genera un fichero JSON con metadatos de las 105 skills, y un fichero `.llms.txt` sirve como índice legible para modelos de lenguaje. El patrón viene directamente de Proto: la documentación es la API.

Nada de esto toca sistemas en producción ni bases de datos. Son definiciones de comportamiento y scripts que se activan cuando el agente los invoca.
