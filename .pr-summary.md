SE-228 Slice 5 adds the loop phasing model for autonomous skills in pm-workspace.

A new rule `docs/rules/domain/loop-phasing.md` defines four maturity levels for any skill that runs autonomous loops: L0 (draft, no execution), L1 (report-only, generates STATE.md and run-log but never commits), L2 (assisted, creates Draft PRs with maker/checker split and loop-budget), and L3 (unattended, requires two weeks of clean L2 history plus explicit human approval). The document includes promotion checklists for L1→L2 and L2→L3, and hard red flags that permanently block promotion (e.g., same verifier and implementer in one session, missing state file, auto-merge without allowlist).

A new script `scripts/loop-phasing-audit.sh` reads the `loop_level` declared in each skill's SKILL.md frontmatter, infers the real level from filesystem evidence (STATE.md, run-log, loop-budget, maker-checker protocol reference), and emits a table with columns `skill | declared | inferred | gap`. Gap is `OK` when declared matches inferred, `OVER` when the skill claims more maturity than its evidence supports, and `UNDER` when evidence suggests a higher level than declared. The script supports `--skill <name>` for single-skill inspection and `--json` for structured output. Exit code is always 0 (informative tool).

The `loop_level` field is added to the SKILL.md frontmatter of the template (L0), overnight-sprint (L2), code-improvement-loop (L2), and tech-research-agent (L1). Each field includes an inline comment pointing to `loop-phasing.md`.

The test suite `tests/test-se228-s5-loop-phasing.bats` covers 34 cases across documentation completeness, script behavior, field correctness, negative/edge cases (nonexistent skill, absent L4, graceful errors, red flag content), and isolation (mktemp + teardown). Score: 89/100, certified.
